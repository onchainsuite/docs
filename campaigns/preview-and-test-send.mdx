Campaigns depend on dynamic wallet traits and merge variables, so most send failures are personalization failures. A preview cycle catches them while they're still cheap.

## Preview

Preview renders the campaign exactly as the send pipeline would, and returns the HTML, the plain-text version, any unresolved merge fields, and any warnings.

```bash
curl -X POST https://api.onchainsuite.com/api/v1/campaigns/$ID/preview \
  -H "x-org-id: $ORG_ID" -H "Content-Type: application/json" \
  -d '{ "recipient": { "email": "reviewer@example.com" } }'
```

Pass a `recipient` to render as a specific contact. Anything you supply is merged over the stored contact, so you can preview edge cases — a contact with no first name, or no ENS — without creating test data.

```json
{ "recipient": { "email": "reviewer@example.com", "firstName": "" } }
```

The response's `missingFields` is the list that will block your launch. Treat a non-empty `missingFields` as a failing check even when the preview looks fine.

## Test send

```bash
curl -X POST https://api.onchainsuite.com/api/v1/campaigns/$ID/send-test \
  -H "x-org-id: $ORG_ID" -H "Content-Type: application/json" \
  -d '{ "to": "reviewer@example.com" }'
```

`to` is required. The same content overrides available to preview work here too.

<Warning>
**Test sends are invisible to analytics.** They carry no tracking pixel and no click wrapping, and record no delivery event. Opening or clicking a test email will never appear in campaign results — and can't be used to verify that tracking works.
</Warning>

Test sends also skip the plan allowance check, so they won't consume your monthly messages.

### Test send errors

| Code | Meaning |
| --- | --- |
| `EMAIL_BODY_EMPTY` | Both HTML and text are blank |
| `EMAIL_PROVIDER_PAYLOAD_INVALID` | A required field is missing — the response lists which |
| `EMAIL_RENDER_VALIDATION_FAILED` | Unresolved merge variables, with a hint on fixing them |

## What to verify

<AccordionGroup>
  <Accordion title="Personalization has fallbacks" icon="user">
    Preview as a contact missing the optional fields. Every variable should degrade to something readable — `Hey there,` not `Hey ,`.
  </Accordion>

  <Accordion title="The sender is the one you chose" icon="paper-plane">
    The sender is resolved from **verified** identities at render time. If the test arrives from an unexpected address, your intended sender isn't verified — check the render warnings and [Sender Verification](/settings/sender-verification).
  </Accordion>

  <Accordion title="Links resolve and carry the right parameters" icon="link">
    Check that UTM parameters appear only where you intended. Existing parameters on a link are never overwritten.
  </Accordion>

  <Accordion title="Rendering on mobile and desktop" icon="mobile">
    Test emails are real emails — check them in an actual client, not just the preview pane.
  </Accordion>

  <Accordion title="The audience is the one you meant" icon="users">
    Re-check the estimate. A segment whose criteria changed since you built the campaign will quietly change your recipient count.
  </Accordion>
</AccordionGroup>

## Ready to launch

A campaign is ready when validation passes with no errors, `missingFields` is empty in preview, a test send renders correctly in a real client, and the audience estimate matches expectation.
