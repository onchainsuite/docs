# V2

TL;DR

* A wallet is the identity. Email is one of N channels, not a prerequisite.

* Multi-channel activation: in-app push (priority 1), email, Telegram, Farcaster, X.

* Protocol Plays library: 8 pre-built, fork-and-edit automations — split into Starter (works on day one) and Advanced (requires data maturity or specific protocol features).

* First-mile rebuilt to deliver one valuable cohort insight in under 10 minutes — with a forward-looking insight template for protocols that don't yet have enough data for a backwards-looking cohort.

* Privacy/ZK remains central wherever a regulated channel is involved. Wallets are public; email, FID, X handle, and similar PII are still bridged via zero-knowledge proofs, GDPR-compliant by architecture.

## **What's changing and why**

The settled 7-category IA stays. The internal engines (R3tain, 3ridge, Onch3n) remain absorbed and invisible to the user. This update reshapes four things on top of that foundation.

| Area | Before | After |
| :---- | :---- | :---- |
| **Identity** | Wallet must be matched to an email before the product is useful. | Wallet is the identity from minute one. Channel-specific resolution happens lazily. |
| **Channels** | Email-only activation. | In-app push, email, Telegram, Farcaster, X — same trigger, fan-out delivery. |
| **Automations** | Blank canvas. User assembles flow from scratch. | Protocol Plays library: pre-wired triggers, segments, copy. User forks and edits. |
| **First-mile** | Connect → set up identity → import audience → build flow → maybe see value. | Connect → auto-index → see one valuable cohort insight in under 5 minutes, before any campaign. |

# **1\.  Wallet-First Identity**

The wallet is the canonical user record. Other identifiers (email, Farcaster FID, X, ENS) are attached to the wallet record as channel handles, present or absent, and resolved lazily when a channel needs them.

### **Build implications**

* **Audience profile schema:** walletAddress is the primary key. email, farcaster, x, fids, and ens are arrays of {value, source, verifiedAt, optInAt}.

* **Drop the requirement that a contact has an email to be “valid.”** A wallet with zero linked channels is a valid contact — it just has fewer reachable channels.

* **Wallet-Email Synthesis stops being a blocking onboarding step.** It becomes a background quality job that incrementally raises match rates as protocols connect X OAuth, upload CSVs, etc.

* **Channel-aware segments.** “reachable on in-app push” is a different filter from “has email.”

### **Privacy & ZK — what changes, what doesn't**

Wallet addresses are public by nature, so wallet-first identity does not soften privacy guarantees. The ZK identity bridge remains central wherever a regulated channel is involved.

| Wallet handles → public, indexed directly. Email / FID / X / Discord handles → still bridged via zero-knowledge proofs. We verify the link to a wallet without storing the underlying PII in a way that exposes it. GDPR-compliant by architecture, opt-in by user. *This is the meaningful split: wallet-first does not weaken our privacy story. It clarifies where ZK matters and where it doesn't.* |
| :---- |

### **Channel resolution table**

| Channel | Identifier needed | Source | Reachability if no extra data |
| :---- | :---- | :---- | :---- |
| **In-app push** | Wallet address only | dApp's connected session | 100% of connected wallets |
| **Email** | Email address | ZK synthesis · CSV · OAuth | Depends on synthesis (target ≥60%) |
| **X** | X profile handle | X OAuth | Depends on synthesis (target ≥60%) |
| **Farcaster** | FID | Wallet → FID lookup (Neynar) | Subset; growing |
| **Telegram** | Channel ID or user opt-in | Bot in protocol's group | Channel-level immediate; user-level requires opt-in |

# **2\.  Multi-Channel Activation**

The Automation flow builder gains channel-specific delivery nodes. Same trigger, same segment, multiple destinations. The Intelligence layer routes a single event to whichever channels are configured and reachable for the wallet.

### **In-app push (priority 1, ship first)**

The fastest channel to ship and the highest-leverage one for retention. No DNS, no SES, no domain warm-up. Build it first.

* **Triggered by:** any on-chain behaviour the protocol cares about (deposit, swap, mint, vote, position-change, liquidation-risk).

* **Delivery surface:** a TS SDK, the protocol drops into their dApp. SDK opens a websocket, OnchainSuite pushes payloads, SDK renders an in-app toast / modal / banner.

* **Identity:** wallet address — already present in the dApp's connected session. No extra identifier needed.

* **Targeting:** same segment engine as email. Trigger and segment definitions are reused; only the delivery node differs.

* **Templates:** minimal HTML / markdown components (toast, banner, modal). Protocol can override styling via SDK theme tokens.

### **Channel build order**

* In-app push — priority 1, ship in v2.0.

* Email — already in build. Continues per existing email-layer doc.

* Telegram channel announcement (bot in protocol's group) — priority 2, v2.1.

### **Build implications**

* **Automation flow node taxonomy:** “Send Email” becomes one of {Send Email, Send In-App Push, Post to Discord, Post to Telegram, Post to Farcaster}. Each is a sibling node type with a shared trigger / segment input contract.

* **Per-channel delivery service in the backend.** Email service stays as-is. Add InAppPushService, DiscordService, TelegramService, FarcasterService — each with the same input contract: { walletAddresses, payloadTemplate, automationId }.

* **Per-channel analytics.** “Sent · Delivered · Opened · Clicked” applies to email. In-app push needs “Delivered · Viewed · Dismissed · Clicked.” Discord/Telegram channels need “Posted · Reactions · Replies.” Analytics schema accommodates per-channel event types.

# **3\.  Protocol Plays Library**

Replace the blank-canvas Automation flow builder for the default new-user experience. New users see a Plays gallery first; the blank builder is the “start from scratch” option, not the front door.

### **Anatomy of a Play**

* **Trigger:** pre-wired on-chain or behavioural event (e.g., “first deposit \> $1k”)

* **Segment:** default rule the protocol can edit (e.g., “all wallets that deposited in the last 24h”)

* **Channel(s):** default channel preference, can be overridden

* **Copy template:** subject lines \+ body with web3 dynamic blocks ({{wallet.last\_position}}, {{chain.gas\_today}})

* **Recommended timing:** delays, send-time logic

* **Peer benchmark:** “protocols using this Play see 28-34% open rate” — drawn from Datum's cross-protocol data

### **Tiers — Starter and Advanced**

Every Play is tagged in the gallery with a tier. The tier determines default sort order, recommendation logic during onboarding, and the “Ready to fork” vs “Needs setup” label shown in the Play card.

| STARTER | Forkable on day one. Works for any protocol with the relevant primitive (a deposit, a stake, a wallet). Doesn't require pre-existing TVL, an active governance system, or a quest framework. Default-recommended in onboarding. |
| :---- | :---- |
| **ADVANCED** | **Requires data maturity, protocol-specific structure, or manual setup.** Example: Governance Vote Reminder needs an active proposals contract; Whale Concierge needs a meaningful TVL distribution; Smart Contract Upgrade Notice is manually triggered. Surfaced in the gallery but never the default recommendation for a freshly-onboarded protocol. |

### **v2.0 Plays — ship all eight**

| Play | Tier | Trigger | Default channel | Steps |
| :---- | :---- | :---- | :---- | :---- |
| **First Deposit Welcome** | **Starter** | First successful deposit to the wallet | In-app push \+ email | 3-step: welcome → educational → ask for feedback |
| **Churn Win-Back** | **Starter** | Wallet inactive 30 / 60 / 90 days | Email \+ in-app on next visit | 3-step ladder by inactivity tier |
| **LP Unbonding Reminder** | **Starter** | Cooldown ends in 24h | In-app push | 1-step: action reminder |
| **Governance Vote Reminder** | **Advanced** | New proposal · 24h before quorum | In-app \+ TG channel | 1-step: reminder with vote CTA |
| **Whale Concierge** | **Advanced** | Wallet crosses $X TVL threshold | Email (white-glove) | 1-step: VIP outreach |
| **Liquidity Migration Alert** | **Advanced** | v1 LP still active 7d after v2 launch | Email \+ in-app | 2-step: notify \+ how-to-migrate guide |
| **Smart Contract Upgrade Notice** | **Advanced** | Manual trigger by the protocol team | Email \+ in-app \+ TG channel | 1-step: signed audit-grade notice |
| **Quest / Referral Invite** | **Advanced** | Active wallet w/o quest enrollment | In-app \+ Farcaster cast \+ TG channel | 2-step: invite \+ nudge |

*Default gallery order: Starter Plays appear first, sorted by relevance to the protocol type detected during onboarding. Advanced Plays are visible but require an explicit “Show advanced” toggle on the gallery for first-week users.*

### **Build implications**

* **Plays live in code:** versioned, typed objects, not user-generated automations stored in DB. A Play is a YAML/JSON spec the Automation engine instantiates as a real, editable automation when forked.

* **Tier is a field on the Play spec.** Adding a new Play requires explicit tier classification at definition time. Tier feeds the gallery's recommendation algorithm and the post-fork “You're ready to ship” vs “You'll need to connect X first” message.

* **Forking a Play** creates a new Automation record initialised from the Play spec. Edits diverge from the spec from that point. Track originPlayId and originPlayTier for telemetry.

* **Peer benchmarks** are computed across protocols (anonymised, opt-in). Backend job aggregates per-play open / click / conversion stats and serves them to the Plays gallery \+ post-send analytics.

* **The Plays gallery** becomes the default landing for the Automations section when the user has zero automations. Show a personalised recommendation order based on the protocol type detected during onboarding.

# **4\.  First-Mile Onboarding**

Goal: protocol sees one specific, valuable insight in under 10 minutes — before they build anything.

### **The 5-step path**

| \# | Step | What happens | Time |
| :---- | :---- | :---- | :---- |
| **1** | **Connect** | Pick chain(s); paste protocol contract address(es). Optionally connect the wallet for self-attribution. | \~30 sec |
| **2** | **Auto-index** | Backend pulls the last 90 days of normalised events. Runs in the background; user moves on. | \~3 min (bg) |
| **3** | **Aha moment** | Auto-generated cohort report shown without user action. Two templates depending on data depth — see below. *One CTA: “Reach these wallets with an in-app push when they next connect” — no email required.* | **at minute 3-5** |
| **4** | **Pick a Play** | From the Plays gallery, fork the recommended Starter Play. Review pre-filled trigger, segment, and copy. Optional: connect TG OAuth, upload email list, install in-app SDK. | \~1 min |
| **5** | **Send test** | Send to a test wallet. In-app push fires immediately if the SDK is installed; email fires if an email is available. | \~30 sec |

### **Aha moment — two templates**

The Aha moment generator picks between two cohort report templates based on data depth at index time. The split is what makes the under-10-minute promise hold for both established protocols and freshly-launched ones — exactly the protocols most likely to sign up for early access.

| Template | When it fires \+ what it says |
| :---- | :---- |
| **Backward-looking (default)** | **Triggered when protocol has ≥100 unique active wallets and ≥30 days of indexed history.** Picks the highest-signal cohort from a fixed library: dormant LPs, whale recent activity, near-churn, high-value not-yet-rewarded, multi-protocol overlap. *Example: “142 dormant LPs from the last 30 days. Median lifetime deposit $4.2k. 38 are also active on Aave.”* |
| **Forward-looking (thin-data)** | **Triggered when the protocol has fewer than 100 wallets or fewer than 30 days of indexed history.** Projects cohort behaviour forward using cross-protocol benchmarks. Frames the insight as a near-future event the protocol can act on now, not a past cohort that's too small to be meaningful. **Example A — Cohort projection:** *“Your earliest 24 wallets reach the 30-day mark on May 28\. At benchmark re-engagement rates for protocols in your category, roughly 14 will go dormant unless reached before then. We can fire the Churn Win-Back Starter Play the moment that 30-day threshold is hit — set it up now and walk away.”* **Example B — Behavioural overlap:** *“Of your 80 wallets, 31 also hold positions on a comparable protocol (cross-checked from public on-chain data). They're already familiar with this primitive — they're your most likely retention save and your most likely referral source. The Quest Play targets them as a starting segment.”* **Example C — Pacing baseline:** *“You're 11 days in. Across 40+ protocols at your stage, the median first-month dropout rate is 38%. Setting up First Deposit Welcome and LP Unbonding Reminder now (both Starter Plays, both work from day one) typically reduces this by 12-18 points.”* |

| Why this matters: the protocols most likely to sign up for early access are pre-PMF or recently-launched. A backward-looking cohort report is impossible for them. The forward-looking template lets us deliver a meaningful Aha at minute 5 even when the protocol has 80 wallets, not 8,000. It also positions OnchainSuite from day one as predictive infrastructure, not just a dashboard. |
| :---- |

### **Build implications**

* **Auto-index pipeline** must handle major DeFi/NFT contract types out of the box (ERC20 transfer, Uniswap-style swaps, Aave/Compound deposit/borrow, ERC721 mint/transfer, governance vote). Use the existing Protocol Normalisation System.

* **Aha-moment generator** is a server-side job that runs once after auto-index completes. Routes to backward-looking or forward-looking template based on wallet count and history depth thresholds (configurable; default ≥100 wallets AND ≥30 days \= backward, otherwise forward).

* **Forward-looking template requires cross-protocol benchmark data** as input. This means the peer benchmark aggregation job (originally scoped to v2.2) needs a v0 in v2.0 — even a coarse “median dropout by protocol type” dataset is enough. Surface this dependency to engineering early.

* **Aha-moment surfaces inline** in the Home dashboard, not as a pop-up. User sees it the moment auto-index finishes.

* **Email/Discord OAuth and in-app SDK install** happen at step 4, not as a gating onboarding requirement.

* **Onboarding telemetry:** track step 1→3 completion rate, time-to-aha, fork rate from Aha moment to a Play, and Aha-template-served (backward vs forward) as a dimension on every funnel metric. These are the north-star metrics for first-mile success.

Suggested order. Each milestone is shippable on its own. v2.0 is split into v2.0a and v2.0b to de-risk the schema refactor and the SDK build. Both sub-milestones ship within the original v2.0 window (Sprints 1-2).

| :---- |

# **Open Questions**

* **In-app SDK distribution:** npm package vs script tag vs both? (Recommend both — npm for dApps with bundlers, script tag for legacy.)

* **In-app push delivery:** websocket vs SSE vs polling fallback? (Recommend websocket primary, polling fallback.)

* **Plays storage:** bundle in repo as YAML, or store in DB seeded at deploy? (Recommend repo \+ seed — versioning is easier in code.)

* **Forward-looking Aha thresholds:** where exactly is the cutover between backward and forward templates? Defaults proposed: \<100 wallets OR \<30 days indexed \= forward. Worth A/B testing once we have enough early access protocols.

* **Peer benchmarks opt-in UX:** explicit toggle in onboarding vs default-on with opt-out? (Recommend default-on, opt-out, anonymised — but flag for legal review and add a clear data-use page accessible from onboarding.)

* **Channel preference precedence** when wallet is reachable on multiple: hard-coded order vs protocol setting vs per-Play override? (Recommend per-Play override with sensible defaults.)

* **Tier upgrade path:** as a protocol matures, should Starter Plays auto-suggest upgrades to Advanced Plays (e.g., “you now have 50 wallets above your whale threshold — set up Whale Concierge”)? Worth scoping for v2.1.

# **Unit Economics Model — 500 Protocols @ 100k Subscribers Each (Monthly)**

Date: 2026-05-26

This document estimates monthly operating costs for hosting **500 protocols**, each with **100,000 subscribers**, sending **100,000 emails per protocol per month**, plus Discord \+ Telegram notifications.

It uses public list pricing where available and fills gaps with explicit assumptions so you can swap numbers easily.

## **1\) Workload Assumptions**

* Protocols (tenants): P \= 500  
* Subscribers per protocol: S \= 100,000  
* Total subscribers: P × S \= 50,000,000 contacts  
* Emails per protocol per month: E\_p \= 100,000  
* Total emails per month: E \= P × E\_p \= 50,000,000  
* Avg email payload billed size (HTML+headers+inline assets, excluding attachments): **100 KB** (0.1 MB) (assumption)  
* Discord \+ Telegram notifications:  
  * Case A (broadcast): N\_p \= 100,000 notifications per protocol **total across both channels** (assumption)  
  * Case B (per channel): N\_p \= 100,000 notifications per protocol **per channel** (assumption)  
  * Total monthly notifications:  
    * Case A: N \= 50,000,000 notifications  
    * Case B: N \= 100,000,000 notifications

## **2\) Cost Drivers (What actually costs money)**

### **A) Email sending (Azure Communication Services Email)**

Azure ACS Email pricing (example doc): **$0.00025 per email sent** and **$0.00012 per MB transferred**.acs\_email\_pricing

**Monthly email cost (send):**

* C\_email\_send \= E × 0.00025  
* With E \= 50,000,000 → C\_email\_send \= $12,500

**Monthly email cost (data transferred):**

* C\_email\_data \= (E × avg\_email\_mb) × 0.00012  
* With avg\_email\_mb \= 0.1 → 50,000,000 × 0.1 × 0.00012 \= $600

**Estimated Azure Email total (monthly):**

* C\_email\_total ≈ $13,100

Notes:

* This excludes inbox/mailbox hosting (e.g., Microsoft 365), dedicated IP warmup, and any third-party deliverability tooling.

### **B) Database (Neon Postgres)**

Neon list pricing (Launch/Scale) includes **$0.35 per GB-month storage** and **$0.106–$0.222 per CU-hour compute** depending on tier.neon\_pricing

At this scale, the DB sizing is dominated by:

* Contacts table size \+ indexes (50M rows)  
* Event tables (email delivery events can become huge if stored per-recipient)  
* Segmentation queries / analytics workloads

**Storage sizing (illustrative, you should measure your real row sizes):**

* Contacts: assume \~ **1 KB per contact row including index overhead** → \~50 GB  
* Delivery events (if persisted per email recipient):  
  * 50M rows/month; even **200 bytes/row** → 10 GB/month (and grows with retention)  
* Other tables \+ indexes \+ bloat \+ PITR history → often **2×–5×** your “raw” table estimate

**Example Neon storage assumption (monthly):**

* DB\_storage \= 400 GB (assumption)  
* C\_db\_storage \= 400 × 0.35 \= $140

**Compute assumption:**

* Let average compute be CU\_avg running 24/7  
* Monthly CU-hours ≈ CU\_avg × 730  
* C\_db\_compute \= (CU\_avg × 730\) × price\_per\_CU\_hr

Example:

* CU\_avg \= 32 CU, price \= $0.222/CU-hr (Scale) → 32×730×0.222 ≈ $5,186

**Estimated Neon total (monthly, example):**

* C\_db\_total ≈ $5,326 (compute \+ storage)

### **C) Queue/Rate-limiting (Upstash Redis)**

Upstash Redis pay-as-you-go list pricing: **$0.20 per 100k commands**, storage **$0.25/GB**, bandwidth free up to thresholds.upstash\_pricing

BullMQ typically uses multiple Redis commands per email job (enqueue, fetch, ack, retry, delayed, etc.). A conservative range:

* commands\_per\_email\_job \~ **10–30** (assumption)

If you enqueue one job per email:

* redis\_commands \= E × commands\_per\_email\_job  
* For E=50M:  
  * Low: 50M×10 \= 500M commands  
  * High: 50M×30 \= 1.5B commands

Cost:

* C\_redis \= (redis\_commands / 100,000) × 0.20  
  * Low: (500M/100k)\*0.2 \= **$1,000**  
  * High: (1.5B/100k)\*0.2 \= **$3,000**

Storage is usually small vs commands unless you store large payloads in Redis.

### **D) App hosting / workers (Render)**

Render list pricing for always-on services (monthly) includes:

* Starter $7, Standard $25, Pro $85, Pro Ultra $450, etc.render\_pricing

At this scale, your biggest compute costs are:

* API servers  
* Background workers (email dispatch, webhooks, inbox polling, etc.)  
* WebSocket gateways

Because instance counts depend heavily on your throughput, here’s a placeholder:

* Assume N\_api \= 6 × Pro Ultra ($450) for API/websocket  
* Assume N\_workers \= 10 × Pro Ultra ($450) for worker fleet  
* C\_render \= (6+10) × 450 \= $7,200

You can model alternatives:

* If you can run on fewer/lower instances, reduce this linearly.

### **E) Frontend hosting (Vercel)**

Vercel Pro starts at **$20/month** (plus usage), and includes **10M Edge Requests/month** and **1TB Fast Data Transfer/month**. Overages start at **$2 per 1M Edge Requests** and **$0.15 per GB** of Fast Data Transfer.vercel\_pricing

There are two very different cost profiles depending on what your “frontend” serves:

**Scenario 1 — protocol operator dashboard only (typical B2B SaaS):**

* Traffic is mostly from a few thousand internal users (protocol teams), not all subscribers.  
* Vercel costs are usually dominated by **seats** (developer seats) and modest bandwidth.  
* Example placeholder: vercel\_monthly \= $200–$1,000 (depends on seats \+ usage).

**Scenario 2 — subscriber-facing web app (50M end-users hitting Vercel):**

* Edge Requests and bandwidth can dominate quickly.  
* Back-of-envelope if each subscriber loads 3 pages/month and each page triggers \~5 edge requests:  
  * Edge requests/month ≈ 50M × 3 × 5 \= 750M  
  * Overage (beyond 10M included) ≈ 740M / 1M × $2 ≈ $1,480  
* If average transferred bytes is 1 MB per subscriber per month:  
  * Data transfer/month ≈ 50M MB \= 50,000 GB  
  * Overage (beyond 1TB included) ≈ 49,000 GB × $0.15 ≈ $7,350

Because your prompt focuses on “subscribers” (contacts) and not “monthly active web users”, treat Scenario 2 as a stress-case unless your product is actually subscriber-facing.

### **F) Discord \+ Telegram notifications**

Discord and Telegram bot APIs don’t usually charge per message. Your main costs are:

* Compute to generate/send messages  
* Egress bandwidth  
* Operational complexity/rate limiting

For cost modeling, treat them as “compute/egress included in hosting” unless you use a paid messaging provider.

### **G) GoldRush (Covalent)**

GoldRush provides multichain data and has plan-based pricing; treat it as a fixed monthly line item unless you have a known “per-credit” cost. The docs list rate limits by plan (e.g., 4 RPS vs 50 RPS).goldrush\_overview

For this model:

* C\_goldrush \= $250/mo (placeholder; replace with your contract)

## **5\) Revenue \+ Profit Margin Scenarios**

Note: if you price email overage close to your provider cost (e.g., $0.0003–$0.0005), your margin will compress quickly because email \+ queueing are the dominant variable costs at this scale.

## **6\) Biggest Hidden Cost Risks (at this scale)**

1. **Storing per-recipient delivery events in Postgres**  
   * 50M rows/month grows extremely fast with retention.  
   * Consider TTL/partitioning or moving high-volume event telemetry to cheaper storage.  
2. **Queue design**  
   * One Redis job per recipient can become expensive and operationally heavy.  
   * Batch jobs (e.g., 1 job \= 1k recipients) dramatically reduce Redis ops.  
3. **Email deliverability**  
   * Deliverability engineering (warming, complaint handling, bounce processing, domain reputation) can add both tooling costs and engineering time.  
4. **Network egress**  
   * Email payloads, webhook callbacks, and analytics export can make egress non-trivial.

# **2\. The Problem We Solve**

**Web3 has a retention crisis.** Brands invest millions in user acquisition only to see a huge % of users churn within 180 days. In some DeFi verticals, churn exceeds 50% after 30 days.

The root cause is not a lack of product-market fit. It is a critical infrastructure gap: **marketing tools are completely blind to on-chain behavior.**

In Web2, when a user buys a product, abandons a cart, or browses a page, brands can immediately reach out with targeted communication and build relationships over time. In Web3, when a user trades $50,000 on a protocol for the first time, the brand cannot speak to them directly. They cannot thank them. They cannot build a relationship. The user simply disappears into the void.

Today, Web3 brands are forced into a “patchwork stack”, stitching together Mailchimp for email, Privy or Dynamic for auth, Dune for analytics, and manual CSV exports in between. This results in fragmented workflows, zero automation, and generic blanket messaging on social media that fails to retain anyone.

# **3\. What is OnchainSuite**

OnchainSuite is the advanced communication layer built natively for Web3 product ecosystems. It creates a seamless bridge between on-chain user behavior and off-chain communication channels, enabling Web3 brands to detect, understand, and respond to user actions on the blockchain through personalized, automated messaging.

The platform operates as a single, unified product with three integrated capabilities working invisibly under the hood.

## **3.1 Core Capabilities**

### **Web3 Customer Data Rail (Identity Resolution)**

*Powered by the Datum Labs proprietary data resolution engine.*

Links anonymous wallet addresses to verified email identities using zero-knowledge privacy-preserving methods. Normalizes user profiles across multiple blockchains and makes consented data usable without compromising security. The PNS also surfaces AI-driven signals, including churn risk scores, high-value cohort identification, and reactivation potential.

### **Marketing Automation Rail (Activation Engine)**

Executes hyper-targeted email campaigns triggered by real-time blockchain events. Provides a visual workflow builder for multi-step automation sequences, A/B testing, template management, and campaign analytics. Brands can send precise communication to users based on any behavior on-chain: mints, swaps, unstakes, deposits, withdrawals, governance votes, all automatically, all personalized.

# **4\. How It Works**

OnchainSuite operates as a continuous, automated loop that replaces the fragmented patchwork stack entirely.

### **The Automated Retention Loop**

**Step 1: Identity Resolution:** A user connects their wallet and email details to a brand’s platform. With OnchainSuiteas the backend, it links the wallet address to a verified email identity using privacy-preserving methods. The user becomes known.

**Step 2: Behavior Detection:** The user performs any on-chain action, trades, stakes, adds liquidity, or mints. OnchainSuite’s analytics engine captures this activity in real time across supported chains and normalizes it into actionable events.

**Step 3: Intelligent Segmentation:** The behavioral data automatically segments users into cohorts: high-value depositors, churn-risk users, dormant wallets, new minters, and more. AI-driven scoring surfaces the most critical segments.

**Step 4: Automated Activation:** Behavior-triggered campaigns fire automatically. A user who withdrew liquidity gets a personalized re-engagement email with an offer (if available). A first-time minter receives a welcome sequence. A high-value trader gets a loyalty reward notification.

**Step 5: Measure & Improve:** Unified analytics track the full loop, from on-chain event to email open to re-engagement. Revenue attribution shows exactly which campaigns drive retention and lifetime value.

| *The entire process is automated, privacy-preserving, and happens in real time. The marketer using ONS never needs to understand the underlying architecture. They simply think, “I want to email everyone who withdrew liquidity this week,” and execute it in one workflow.* |

# **2\. Company and Product**

## **2.1 What OnchainSuite is**

OnchainSuite is a single, unified product that operates as a continuous, automated retention loop: detect on-chain behaviour, resolve identity, segment, message, and measure. To the user, a Web3 growth marketer who is not a developer, it presents seven task-centric sections: Home, Campaigns, Automations, Audience, Analytics, Integrations, and Settings. The interface is deliberately benchmarked against Klaviyo's design philosophy: enormous backend complexity hidden behind an intuitive surface.

The platform rests on four product pillars.

### **Wallet-first identity**

The wallet is the canonical user record. Other identifiers (email, ENS, and chat handles) attach as optional channel handles and are resolved only when a channel needs them, using privacy-preserving, zero-knowledge methods. A wallet with zero linked channels is still a valid contact; it simply has fewer reachable channels. In-app push reaches 100% of connected wallets with no extra identifier, which is why it leads the channel set. This removes the single biggest source of onboarding friction: a protocol gets value from minute one, without first matching wallets to emails.

### **Multi-channel activation**

A single trigger and segment fan out to whichever channels are configured and reachable for each wallet. The automation builder uses channel-specific delivery nodes that share a common trigger and segment contract, so adding a channel never means rebuilding a campaign. In-app push, delivered via a lightweight JS SDK drops into its dApp and is the highest-leverage channel: it needs no DNS, no email warm-up, and no extra identity resolution. Email is live today; chat channels follow on the roadmap (Section 10).

### **Protocol Plays library**

A library of pre-built, fork-and-edit automations replaces the blank canvas as the default new-user experience. Plays such as First Deposit Welcome, Liquidity Migration Alert, Governance Vote Reminder, Whale Concierge, Churn Win-Back, Smart Contract Upgrade Notice, and LP Unbonding Reminder ship with pre-wired triggers, editable segments, copy templates with web3 dynamic blocks, and peer benchmarks drawn from anonymised cross-protocol data. New users see the Plays gallery first; the blank builder becomes the “start from scratch” option, not the front door. A non-technical marketer launches a sophisticated retention campaign in minutes rather than weeks.

### **Sub-10-minute first-mile**

Onboarding is the company's clearest competitive wedge. A protocol connects its contract addresses, the backend auto-indexes 90 days of normalised events in the background, and an auto-generated cohort report appears, for example: “142 dormant LPs from the last 30 days; median lifetime deposit $4.2k; 38 are also active on Aave.” The single call-to-action requires no email match. This delivers a quantified, valuable insight before the user has built anything: the immediate value that, in a founder-led sales motion, converts demos into deals.

Underpinning all four pillars is a proprietary **Protocol Normalisation System** that standardises events across Ethereum, Solana, Base, and Polygon, so that a “liquidity deposit” looks identical whether it happens on Uniswap, Aave, or Aerodrome, and surfaces AI-driven signals such as churn-risk scores and high-value cohort identification.

## **2.3 Defensibility**

* **Protocol Normalisation System:** the proprietary cross-protocol, cross-chain event schema is the core moat. It is the difference between a marketer thinking “email everyone who removed liquidity this week” and executing it in one workflow, versus maintaining brittle Dune queries per protocol.

* **Cross-protocol behavioural data:** anonymised, opt-in peer benchmarks compound with every protocol that joins, creating a data network effect competitors cannot replicate without scale.

* **Privacy-by-design identity bridge:** the ZK-based linkage is both a feature and a regulatory hedge (see Section 9).

Surgence is a go-to-market partner to more than 40 blockchain companies, and OnchainSuite has a signed agreement to integrate into Surgence's partner base at launch. ## **5.2 Convert the design partners (now → month 3\)**

The most important near-term task is turning Moonwell and Surgence into named reference customers with quantified outcomes, for example, “Moonwell increased 30-day depositor retention from X% to Y%.” Without this evidence, every later step is harder; with it, fundraising, grants, and sales all compound. Because the founders work inside both Surgence and Moonwell, these conversions start from a position most startups never have.

The company will codify additional design-partner agreements with explicit transition terms. Industry best practice converts 60–80% of design partners to paid when the agreement defines transition pricing and timeline up front, and below 30% when it does not. The standard incentive structure:

* Usage credits during the design phase, plus a discount on the first year of paid usage.

* Twelve months of early access to advanced features before general availability.

* A small advisory-equity or revenue-share arrangement for the most strategic partners.

# **6\. Business Model and Pricing**

## **6.1 Revenue model**

OnchainSuite is a SaaS business. Revenue is roughly 85% recurring subscriptions, 10% professional services (custom integrations and white-glove onboarding for larger protocols), and 5% partner revenue share (agency and white-label arrangements). The centre of gravity is recurring subscription revenue with expansion as protocols grow their reachable wallet base and add channels.

## **6.2 Pricing model**

Pricing is modular and usage-based: a protocol pays for what it actually uses, so the price scales smoothly from the smallest project to the largest, with no rigid tiers or hard caps. The bill is built from three components, each carrying the company's margin, plus a small base fee:

* **Email subscribers:** the protocol's emailable audience. Each subscriber bundles a monthly send allowance (10 sends per subscriber), so sending capacity scales automatically with list size.

* **Tracked wallets:** the number of on-chain wallets the protocol monitors. This is the platform's distinct value, independent of email, and is charged separately.

* **Base fee:** a small flat fee covering a share of fixed platform overhead.

Channels are email and in-app push first, with Telegram and Discord on the roadmap; the product does not offer SMS. Email sending runs on Azure at launch, migrating to AWS SES over time to lower per-email cost and add dedicated-IP deliverability control.

Because pricing is continuous, the figures below are illustrative reference points at a 60% target margin rather than fixed packages:

| Profile | Subscribers | Tracked wallets | Indicative price/mo |
| :---- | :---- | :---- | :---- |
| Small | 500 | 500 | \~$45 |
| Growing | 1,000 | 2,000 | \~$140 |
| Mid | 10,000 | 10,000 | \~$660 |
| Large | 50,000 | 50,000 | \~$3,300 |

*Illustrative, at a 60% margin. Price scales continuously with usage; larger protocols move to custom agreements. An internal calculator generates an exact quote from any subscriber and wallet count.*

## **6.3 Cost structure**

Costs fall into two buckets. Fixed platform infrastructure is a modest shared monthly base; variable costs scale per customer with usage. All infrastructure is funded from the company's own revenue and the raise, not by Datum Labs.

Variable cost per customer is small: roughly $0.0001 per email send (AWS SES / Azure), a fraction of a cent per wallet for storage, and GoldRush data at about $0.025 per tracked wallet per month. **GoldRush is the largest variable cost driver**, which is why wallet tracking, not email, is the cost and value centre of the product. A single GoldRush Professional plan safely supports roughly 10,000 tracked wallets across all customers; the company provisions additional plans before that ceiling to prevent any service interruption.

## **6.4 Margin and break-even**

Every customer is priced to a **60% gross margin** by design, and the margin holds across the full size range because the model marks up actual cost. The fixed infrastructure base is small enough that the company **covers its entire fixed infra cost with roughly five paying customers**; every customer beyond that contributes margin straight to the bottom line. This is the structural advantage of a software product on a shared data pipeline: each additional protocol adds far more revenue than cost.

# **9\. Risks and Mitigations**

We present the risks a sophisticated crypto and SaaS investor will probe, with the company's mitigation for each. Surfacing these directly is a deliberate choice; diligence rewards candour.

| Risk | Sev. | Mitigation |
| :---- | :---- | :---- |
| **Wallet-as-personal-data (GDPR)** | High | Keep channel linking strictly opt-in with explicit consent; store linkage off-chain and deletable; document a DSAR/erasure workflow; lead with the ZK identity bridge as privacy-by-design (see 9.1). |
| **Customer concentration** | Med | Mitigated by the signed Surgence channel (40-plus companies) and the Moonwell relationship; the first cohort comes from a diversified partner base, not a single account. |
| **Platform / data dependency** | Med | Reliance on Alchemy, Neynar, Dune, and chain RPCs is hedged with a redundant data source per chain and self-hosted nodes for the top two chains. |
| **Email deliverability for crypto** | Med | Crypto email is heavily filtered; mitigate with SES warm-up, dedicated IPs, and strict SPF/DKIM/DMARC, and lean on in-app push, which sidesteps the issue entirely. |
| **Market timing / “why now”** | Med | Holder closing, Spindl’s acquisition, XMTP’s Series B and Push’s pivot show a category in flux; the retention layer is the open frontier. |
| **Moat durability** | Med | Quantify the PNS’s integration-time savings versus per-protocol Dune queries; compound the cross-protocol benchmark data network effect. |
| **Budget cyclicality** | Med | Crypto marketing budgets swing with market cycles; favor annual prepay contracts and anchor on in-app push, the lowest-cost, highest-retention channel. |

## **9.1 The privacy question, addressed directly**

In 2025 the European Data Protection Board adopted guidelines treating blockchain identifiers, including public keys and wallet addresses, as data that can indirectly identify a person, and reaffirmed that technical immutability does not excuse non-compliance with rights such as erasure. **This is the most important regulatory question for any wallet-identity business, and the company treats it as a first-class design constraint, not an afterthought.** OnchainSuite’s architecture is built for it: channel handles are linked off-chain in deletable storage, linking is consent-gated, and the zero-knowledge identity bridge minimises exposed data by design. The use-of-funds reserve includes a formal privacy/legal review before scaling into EU markets.

*Source: EDPB Guidelines 02/2025 on processing personal data through blockchains (adopted April 2025; consultation closed June 2025; final text pending). Also, CNIL guidance, William Fry and Lexology analyses.*
